# Tokenkeeper v0.1.0 Post-release Review

**Review date:** 2026-07-12  
**Reviewed source:** `main` at `c783916ddda709bbf3b3e769eb7e64d95bf25ac2`  
**Published tag:** `v0.1.0` at `f3ef34015f5379c90d07362337f8cf69887edd00`  
**Method:** PRD/Workplan traceability, source and test review, native macOS filesystem reproductions, Rust quality gates, coverage diagnostics, Git/release inspection, and Homebrew formula checks.

## Executive Verdict

Tokenkeeper has a compact dependency-free Rust core, conservative remediation behavior, useful Unix permission checks, and clear security documentation. However, the `v0.1.0` acceptance claim is not supported end to end. The audit reproduced unsafe states reported as `PASS`, showed that the macOS ACL adapter does not understand native ACL text, found major omissions in the required finding report, and established that the published tag is not an ancestor of the current `main` branch.

`v0.1.0` should therefore be treated as requiring post-release hardening. The existing tag must remain immutable; fixes should be released under a new version after the blocking tasks below pass native integration and distribution gates.

## Scope and Positive Evidence

- The architecture follows the intended `CLI ŌåÆ profiles ŌåÆ resolver ŌåÆ inspector/ACL ŌåÆ report` flow.
- Production code has no third-party Rust dependencies, network client, target-content read, remediation subprocess, or automatic filesystem mutation.
- Exact-path validation rejects parent traversal and detects target/intermediate symlinks inside declared relative paths.
- Unix owner, node-type, and mode-bit policy checks work for the covered fixtures.
- Current `main` conservatively suppresses remediation for known owner, node, symlink, ancestor, and target ACL findings.
- Linux is documented as incomplete and the Rust core builds in Linux CI.
- Configured tests, format, and Clippy gates pass; no tests are ignored.

## Blocking and High-severity Findings

### TK-REV-001 ŌĆö Writable ancestor ACL produces a false `PASS` (Blocker)

ACL evaluation is invoked only for the target in [`src/inspector.rs`](../src/inspector.rs). The ancestor loop checks symlink and Unix `0o022` bits, ignores metadata errors, excludes the trusted Home itself, and does not evaluate owner, directory node type, or ACL rights.

Native reproduction: a `0600` credential below a `0700` directory with `everyone allow write`/`add_file` ACL was reported as `PASS` with exit code `0`. A non-owner principal can still replace or create configuration through that parent.

Required direction: collect a complete observation for every component from canonical Home through the target; treat foreign ownership, replacement-capable ACL rights, unexpected node type, and unreadable metadata as finding or incomplete, never clean.

### TK-REV-002 ŌĆö Credential-bearing executable configs lose confidentiality (Blocker)

Research classifies Codex `~/.codex/config.toml` and `~/.aws/config` as credential-bearing and executable/integrity-sensitive. Built-in profiles select only `ExecutableConfig`, whose mode policy prohibits foreign write but allows foreign read.

Native reproduction: a Codex config with mode `0644` was reported as `PASS`. This contradicts the PRD rule that any known credential-bearing config uses `CredentialConfig` confidentiality regardless of extension.

Required direction: immediately apply the stricter credential policy to these locations; then replace the single enum choice with composable confidentiality, integrity, node, ancestor, and ACL requirements.

### TK-REV-003 ŌĆö Native macOS ACL format is not parsed (High)

The parser in [`src/acl.rs`](../src/acl.rs) expects synthetic entries such as `user:alice:allow:read`. Native `acl_to_text()` output contains a `!#acl 1` header and entries such as:

```text
group:<UUID>:everyone:12:allow:read
```

A `0600` credential with `everyone allow read` therefore returns `UNKNOWN` rather than the required `FINDING`. Unit tests in [`tests/acl.rs`](../tests/acl.rs) mirror the synthetic grammar and never exercise the macOS filesystem adapter.

Required direction: prefer structured ACL entry/tag/qualifier/permission APIs; otherwise implement the complete native grammar and permission vocabulary. Validate with real temporary ACLs on macOS.

### TK-REV-004 ŌĆö ACL acquisition errors can become clean state (High)

`acl_get_file() == NULL` is treated as ŌĆ£ACL not presentŌĆØ without examining `errno`. Permission denial, unsupported filesystem behavior, allocation failure, target disappearance, and other backend errors can therefore become `NotPresent` and then `PASS`.

Required direction: distinguish a confirmed empty ACL from every acquisition failure; failures become `UNKNOWN/INCOMPLETE`. Add injected/error-path tests and anchored descriptor-based inspection where practical.

### TK-REV-005 ŌĆö FR-05 finding/report contract is substantially absent (High)

The renderer prints status, path, Rust `Debug` reasons, and an optional `chmod`. It does not provide stable rule ID, severity, current metadata, expected policy, risk, manual-guidance explanation, or the full checked scope. `UNKNOWN` reasons and `AccessDenied` messages are stored but not rendered. Modes appear as decimal debug values such as `420`.

Required direction: introduce a structured public finding model and golden CLI output tests. Preserve known findings together with incomplete ACL state rather than forcing one mutually exclusive result.

### TK-REV-006 ŌĆö Trusted Home is controlled by `$HOME` (High)

The binary accepts any absolute directory owned by the current UID as trusted Home. A substituted Home containing none of the profiles produces only `SKIP` results and exit code `0`, while the real account Home is never checked. Elevated/root invocation also lacks an explicit policy.

Required direction: resolve Home from the OS account database using the intended UID, compare `$HOME` only for a warning, reject or explicitly model elevated execution, and include canonical audited Home/scope in the report.

### TK-REV-007 ŌĆö Depth limits return partial success (High)

Traversal returns success immediately at `max_depth` even when a directory may contain uninspected descendants. The current resolver test creates a deeper file and expects the shortened result. PRD acceptance requires a limit breach to produce `INCOMPLETE`, not partial `PASS`.

Required direction: detect descendants beyond the depth budget and fail closed. Track completeness separately from findings.

### TK-REV-008 ŌĆö Published and current `0.1.0` are different source trees (High)

The `v0.1.0` tag points to `f3ef340`, current `main` points to `c783916`, and the tag is not an ancestor of `main`. Current `main` contains security-relevant fixes absent from the release while both binaries report version `0.1.0`.

Required direction: never move the existing tag. Complete the hardening backlog, bump the package version, tag the reviewed `main` lineage, verify the archive checksum, and publish a new formula/release with traceable evidence.

### TK-REV-009 ŌĆö Homebrew lifecycle acceptance was weakened (High)

The canonical Workplan requires source install, audit, test, upgrade, and uninstall validation. The archived task allowed a documented blocker and was marked `PASS` even though install/test lifecycle evidence was incomplete. Current style, audit-by-name, and minimal `brew test` pass, but clean source reinstall, upgrade, and uninstall were not executed by this read-only review and remain absent from CI.

Required direction: add a release workflow that exercises the full supported lifecycle on clean macOS runners and stores the exact tag/archive/formula identity.

## Medium and Low Findings

### TK-REV-010 ŌĆö Semantic roots are not walked from Home component by component

`MacApplicationSupport` is assembled as `Home/Library/Application Support` before no-follow checks begin. A symlink at `Library` can be followed by `symlink_metadata()` before it is diagnosed, and an absent optional target may become `SKIP` rather than a symlink finding.

### TK-REV-011 ŌĆö Entry-count budget does not bound enumeration memory

`read_dir()` is fully collected and sorted before recursive `max_entries` checks. A large directory can consume unbounded time and memory even with a tiny configured limit.

### TK-REV-012 ŌĆö CLI scope combinations are ambiguous

The parser accepts `--profile` together with `--path/--policy`; the custom path branch returns before profiles are validated or scanned. Duplicate selectors are also not normalized, and an all-optional scope can return exit `0` with zero checked targets.

### TK-REV-013 ŌĆö Platform and availability behavior is hardcoded

`profiles` prints `macOS/Linux` for every profile and omits availability. Runtime selection checks `Platform::MacOs` even on other platforms. `XdgConfig` ignores a safe `XDG_CONFIG_HOME` inside canonical Home. Linux library results can print per-target `PASS` while the CLI only compensates with global exit `2`.

### TK-REV-014 ŌĆö Profile evidence and fixtures are incomplete

Evidence is stored once per profile rather than per location. The research document records a global date/platform but no verified runtime versions and no row-level source mapping. Built-in tests do not freeze the complete expected location inventory, evidence state, platform availability, or policy composition. Runtime also relies on CI tests rather than validating the embedded registry before use.

### TK-REV-015 ŌĆö Security verification does not meet the PRD strategy

Missing coverage includes native ACL FFI, ancestor ACL/ownership, access denied, depth incomplete behavior, canonical Home mismatch, root/sudo, actual ESC bytes, full CLI exit codes, golden reports, exact profile fixtures, and proof that target contents are not opened. Network/subprocess absence is supported by static source inspection rather than a dedicated enforcement test.

### TK-REV-016 ŌĆö Coverage and CI hardening are incomplete

The review threshold in `SPECS/COMMANDS/REVIEW.md` is 80%; measured line coverage is 71.08%. CI runs fmt/test/Clippy but not coverage, explicit build/check, Homebrew lifecycle, or native ACL fixtures. Cargo commands omit `--locked`; the Rust toolchain floats on `stable`; `actions/checkout` uses a movable major tag.

### TK-REV-017 ŌĆö Terminal sanitization is not applied to all user input

Filename display sanitizes control characters, but unknown CLI arguments and profile IDs are interpolated directly into errors. The existing ŌĆ£ansiŌĆØ test contains the word `ansi`, not an ESC byte.

### TK-REV-018 ŌĆö Package and specification lifecycle metadata need reconciliation

`Cargo.toml` lacks `license` and `rust-version`. The PRD remains `Draft`, contains resolved release/license/tap questions, and does not define precedence for paired policies or the semantics of an all-optional clean scope. Release, formula, source, and documentation version state need one authoritative contract.

## Root-cause Analysis

1. Task-local acceptance criteria drifted from the canonical Workplan, allowing documented external blockers to be archived as `PASS`.
2. Tests were derived from implementation-shaped fixtures rather than native boundary contracts, especially for macOS ACL text.
3. Reviews emphasized green unit gates without an end-to-end PRD-to-native-fixture traceability matrix.
4. The release tag was created on feature history before the squash merge, and later fixes retained the same package version.
5. The domain model compresses independent confidentiality, integrity, completeness, evidence, and reporting concepts into single enums or mutually exclusive results.

## Release-hardening Roadmap

1. Eliminate all false `PASS` paths in ACL, ancestors, credentials, Home identity, and traversal completeness.
2. Replace synthetic platform assumptions with native integration fixtures and structured result/report models.
3. Complete profile provenance, CLI/platform behavior, coverage, and supply-chain/release CI.
4. Re-run the full PRD acceptance matrix and publish an immutable successor release from reviewed `main` lineage.

Success requires zero false `PASS` in the reproduced scenarios, explicit `UNKNOWN` for every incomplete backend operation, 100% acceptance-scenario traceability, coverage at or above the project review threshold, and a release tag/formula/archive that all identify the same reviewed source tree.
